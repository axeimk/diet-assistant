# マイグレーションはコード内に持ち、`diet init`で適用する

- ステータス: 承認済み
- 日付: 2026-07-21

食塩相当量の列追加で初めてスキーマ変更が必要になった。マイグレーションSQLは
`db.py`の`MIGRATIONS`（版番号→SQL）に置き、`PRAGMA user_version`より新しいものだけを
`diet init`が順に適用する。`migrations/*.sql`は履歴の記録であり実行されない。

## 検討した代替案

- **`migrations/*.sql`をディスクから読んで実行する** — ファイル名が正本らしく見えるが、
  `migrations/`はパッケージに含まれず（`src`レイアウトでリポジトリ外にインストールされる）、
  `pip install`後に実行時のパスが解決できない。SQLをコード内に置けばこの問題が消える。
- **Alembicを導入する** — 1人用ローカルツールに対して依存と概念が重い。標準`sqlite3`だけで
  済ませるという既存の選択（`docs/design.md`）と一貫しない。
- **`diet migrate`を独立コマンドにする** — 覚えるコマンドが増える。`init`はもともと冪等な
  セットアップ手順であり、そこに寄せるほうがCLI表面が小さい。`doctor`が版ずれを検出したとき
  `init`を案内する。

## 結果

- 新規DBは`SCHEMA_SQL`から作られ`user_version = SCHEMA_VERSION`が直接入るため、
  マイグレーションは実行されない。既存DBかどうかは`meals`テーブルの有無で判定する。
- マイグレーションは冪等ではない。適用済み判定は`user_version`だけに依存するので、
  手でスキーマをいじるとずれる。
- ダウングレードは持たない。取り消しはバックアップからの復元で行う。
