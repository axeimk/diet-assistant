# record-meal — 食事を記録し、検証とフィードバックまで行う

食事の報告を構造化JSONにして `diet meal add` で登録し、読み戻して検証したうえで、
目標に対する進捗と短いフィードバックを返す。

## 手順

1. 画像・撮影日時・ユーザーの補足文を確認する。
2. 食品候補と量を範囲で推定する。パッケージ表示・公式栄養表示・ユーザー申告値があれば
   画像推定より優先する。
3. 大きな不確実性があるか判定し、結果を大きく変える場合だけ質問する。
   小さな不明点は合理的に推定し、仮定を `text` または品目の `note` に書き残す。
4. ルーティーンを照合する（下記「ルーティーン確認」）。
5. `meal_type`・日時・食事全体の栄養範囲・確信度・品目・仮定を構造化JSONにする
   （下記「構造化JSONの形式」）。
6. `diet meal add --json <file>` で登録する。SQLを直接実行しない。
   JSONファイルは一時ディレクトリに書く（リポジトリ内に残さない）。
7. `diet meal show <id>` で読み戻し、日時・区分・範囲・品目を検証する。
   戻り値だけを信じず、必ず読み戻す。
8. `diet report daily --date <レポート日> --format json` と
   `diet advice today --date <レポート日>` を
   実行し、フィードバックを組み立てる（下記「記録後のフィードバック」）。

## ルーティーン確認

`config/profile.json` の `routine`（実施順のステップ列。例:
`["weight", "breakfast", "lunch", "snack", "dinner", "report"]`）と当日の記録を照合する:

1. `diet report daily --date <レポート日> --format json` で同じレポート日の記録を取得する。
   `config/profile.json` の `day_start_time` より前の日時は、暦日の前日をレポート日とする
   （手順8と同じコマンド。登録前に一度実行しておくと照合と重複確認を兼ねられる）。
2. 今回の報告がルーティーン上のどのステップかを特定し、それより前のステップで
   当日の記録が無いものを探す。対応: `weight` → `metric.weight`、
   `breakfast`/`lunch`/`snack`/`dinner` → `meals` の `meal_type`、`exercise` → `exercises`。
3. 抜けがあれば、記録とフィードバックを済ませたあとに**一言だけ**確認する
   （例: 「朝食はどうでしたか？」）。複数抜けていてもまとめて1回で聞き、詰問調にしない。
   食べていない・測っていないという返答はそれで完結とし、記録を強要しない。
   ただし `snack` は任意ステップで、食べない日があるのが普通なので
   抜けていても確認しない（記録されたときにルーティーン上の位置が分かるだけでよい）。

`routine` が未設定なら何もしない。`report` ステップの扱いはwrite-reportスキル側の仕事。

## 構造化JSONの形式

```json
{
  "eaten_at": "2026-07-21T12:35:00+09:00",
  "meal_type": "lunch",
  "text": "ご飯は大盛り。唐揚げ5個。マヨネーズ少し。量は写真からの推定。",
  "estimated_calories": 850,
  "calories_min": 700,
  "calories_max": 1000,
  "protein": 32,
  "fat": 35,
  "carbohydrates": 100,
  "fiber": 4,
  "sodium": 3.4,
  "estimation_confidence": "medium",
  "source": "claude-code",
  "items": [
    {"name": "ご飯", "amount_text": "大盛り", "fiber": 0.6, "confidence": "medium"},
    {"name": "唐揚げ", "amount_text": "5個", "fiber": 0.4, "confidence": "high"}
  ]
}
```

- `sodium` は食塩相当量（g）。ナトリウム量（mg）ではない。
- `fiber` は食物繊維（g）。日本の栄養成分表示には食物繊維の記載がないことが多いが、
  その場合も食材構成から必ず推定して入れる（未記入にしない）。飲料・卵・肉など
  ほぼ含まれない食事は `0` を明記する。品目にも `fiber` を記録できる。
- `estimation_confidence` と品目の `confidence` は `low` / `medium` / `high`。
- `source` は記録経路の識別子（自分がどの環境かに合わせる。例: `claude-code`）。
- 既存記録の修正に使う更新用JSONは変更する列だけを書く（全列を書き直さない）。

## 記録後のフィードバック

登録結果の説明に続けて、この3点を短く返す（合計5行程度まで）:

1. **登録内容と仮定**: 何をどう推定したか。確信度が `low` の品目はそう言う。
2. **目標に対する現在地**: `advice today` の `evidence`（摂取・目標・残り）を数値で示す。
   目標未設定・プロフィール不足でCLIが計算できないときは、その旨だけ伝える。
3. **一言評価と助言**: `advice today` の `priority_action` を軸に1つだけ。
   P/F/C・食塩に目立つ偏りがあれば事実として添える。順調な日はまずそれを認める。
   残り摂取可能カロリーが少ない・超過している日でも、極端な制限や翌日での相殺は
   提案しない（7日以上の傾向で判断する）。

## 注意点

- 本番データへの記録が目的なので `--root` は付けない（`--root` を使うのはコード変更の
  検証時だけ）。
- 当日すでに同区分の記録があるときは、重複登録の前にユーザーに意図を確認する
  （間食の複数回は正常。昼食2件は聞き直す価値がある）。
- 日時はユーザーの申告・写真の撮影日時を優先し、無ければ現在時刻。タイムゾーンは
  ローカル（記録の `eaten_at` とレポートの `--date` がずれると当日集計に出なくなる）。
