# Diet Assistant Rules

SQLiteを正本に食事・運動・体重・目標を長期記録し、CLIとAIエージェントから傾向分析と助言を得る個人向けツール。

## コマンド

- 既存の`.venv`がある場合、作業開始時から仮想環境内のコマンドを使う。シェル間で`source`の状態が引き継がれない場合は、最初から`.venv/bin/diet`、`.venv/bin/pytest`などを直接実行し、システムのPATH上のコマンドを先に試さない。
- セットアップ: `uv venv --python 3.14 && source .venv/bin/activate && uv pip install -e '.[dev]'`
- テスト: `pytest`
- lint: `ruff check .`
- 型チェック: `basedpyright`（recommendedモード。警告でも失敗扱い）
- 起動: `diet <サブコマンド>`（動作確認の手順はverifyスキルを参照）

## 作業の進め方

- コードを変更したら、報告の前に`pytest`・`ruff check .`・`basedpyright`を回す。
- 完了報告（コミットする場合はコミット）の前に1回、verifyスキルのレシピに従って実際に`diet`を起動・操作し、変更が動くことを確認する。検証は必ず`--root <一時ディレクトリ>`で行い、本番の`data/diet.db`を触らない。
- 上記2項は「プログラムの動作に影響する変更」が対象。ドキュメント・エディタ設定・コメントのみの変更では行わない。
- 検証結果は成否をそのまま報告する。テストが失敗したら失敗と言う。伏せたり言い換えたりしない。
- 破壊的な操作（記録の削除、写真の`--apply`削除、履歴の書き換え、外部への送信）は実行前に確認を取る。
- 同種の作業を繰り返していると気づいたら、スキル化を提案する（スキルは`.agents/skills/`に置き、`.claude/skills/`からsymlinkする）。

## 設計記録

設計上の決定は`docs/adr/`に記録する。記録する基準と形式は`docs/adr/README.md`に従う。

## 応答と実装

- 回答と生成ドキュメントは日本語で書き、結論を先に述べる。
- コード中の識別子は英語で書く。
- Pythonコードは型ヒントを付け、Basedpyright recommendedモードを成功させる。
- 正式な記録はSQLiteを正本とする。会話履歴やAIの記憶を正本にしない。
- DBは原則として`diet` CLIまたは専用スクリプト経由で変更する。
- ユーザーの記録を黙って修正しない。変更時は対象、変更内容、理由を説明する。

## 推定と助言

- 食事写真だけで食品や量を断定しない。
- 推定値には最小値、最大値、代表値、`low`・`medium`・`high`の確信度を持たせる。
- `sodium`は食塩相当量（g）で記録する。ナトリウム量（mg）ではない。公式表示の「食塩相当量」をそのまま使う。
- 不明点が小さい場合は合理的に推定し、仮定を`note`または品目の`note`に記録する。
- 大きな誤差につながる場合のみユーザーに確認する。
- パッケージ記載値、公式栄養表示、ユーザー申告値を画像推定より優先する。
- 1日単位ではなく7日、14日、28日の傾向を重視する。
- 単日の超過を翌日の極端な制限で相殺せず、過度な食事制限を提案しない。
- 運動消費カロリーを過大評価しない。
- 行動変更は原則1つを最優先にし、代替案を添える。
- 計画変更時は理由と数値根拠を示す。
- 医療的な診断や治療判断をしない。体調不良や疾病に関する判断は専門家への相談を促す。

## プライバシーと安全

- 個人データ、写真、DB、バックアップ、生成レポート、`config/profile.json`、`.env`をGitへ追加しない。
- APIキーをコードやログへ出さない。デフォルトで外部サービスへ個人データを送らない。
- 写真削除は先に`diet photo cleanup`でdry-runし、明示的な承認後だけ`--apply`する。
- DB更新にはトランザクションを使い、バックアップにはSQLite Backup APIを使う。

## 定型作業

- 食事の記録（写真・テキスト・申告のいずれも）はrecord-mealスキルの手順に従う。
- 日次・週次レポートの作成はwrite-reportスキルの手順に従う。
- 記録やレポートの依頼を受けたら、`config/profile.json`の`routine`（実施順のルーティーン）と当日の記録を照合し、抜けている先行ステップがあれば一言だけ確認する。詰問調にせず、「していない」という返答はそれで完結とする。`snack`は任意ステップなので、抜けていても確認しない。
